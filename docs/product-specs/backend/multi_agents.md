# 后端规格说明书: 多智能体架构与 Prompt 设计 (Multi-Agent & Prompts)

本规格说明书定义了“多智能体（Multi-Agent）系统”中各智能体的职责分工、边界接口以及提示词集中管理的机制。

---

## 1. 多智能体架构职责分布

每个智能体都被定义为一个专门化的 Python 类，内建大模型客户端的调用接口，并共享全局的 `ResearchState` 用于状态与日志的自动追加记录。

```
          +-------------------------------------------------------------+
          |                       Planner Agent                         |
          |  - 生成子方向与子查询词                                       |
          |  - 根据深度 facts 判断是否进一步向下挖掘                      |
          +------------------------------+------------------------------+
                                         |
                                         v
          +-------------------------------------------------------------+
          |                       Searcher Agent                        |
          |  - 网页检索路由器 (Web vs Academic)                          |
          +------------------------------+------------------------------+
                                         |
                                         v
          +-------------------------------------------------------------+
          |                       Scraper Agent                         |
          |  - 正文网页抓取与 HTML 降噪清洗                              |
          +------------------------------+------------------------------+
                                         |
                                         v
          +-------------------------------------------------------------+
          |                       Extractor Agent                       |
          |  - 事实归纳与断言要点提炼                                    |
          +------------------------------+------------------------------+
                                         |
                                         v
          +-------------------------------------------------------------+
          |                      Synthesizer Agent                      |
          |  - 事实汇总去重、章节填充与 Inline 引用生成                 |
          +-------------------------------------------------------------+
```

### 1.1 规划智能体 (Planner Agent)
* **核心类名**: `PlannerAgent`
* **职责**:
  1. **大纲拆解**: 解析用户的研究总主题，结合最大广度参数 `breadth`，生成一系列互不重复的细分子方向。为每个子方向生成 2-3 个最具针对性的搜索引擎检索关键词（Queries）。
  2. **递归评估**: 在非末尾深度时，阅读 `ExtractorAgent` 已在该子方向下搜集到的全部事实，推导是否还需要对某些未探明的信息盲区追加新的深度检索词。

### 1.2 检索路由器 (Searcher Agent)
* **核心类名**: `SearchRouter`
* **职责**:
  1. **意图路由**: 根据 Query 中的自然语言含义，自动判定是否路由至**学术文献搜索**（arXiv 与 Semantic Scholar）还是路由至**网页通用搜索**（Tavily Web Search）。
  2. **并发检索**: 在 `all` 或复合模式下，并发启动多路搜索通道，进行文献与新闻数据的综合合并，并进行 URL 级别的查重。

### 1.3 网页抓取清洗 (Scraper Agent)
* **核心类名**: `ScraperAgent`
* **职责**:
  1. **动态解析**: 优先尝试使用 Jina Reader API 对目标 URL 进行转化，可直接获取剔除广告/脚本排版良好的 Markdown。
  2. **本地降噪**: 在 Jina API 被限流或网络故障时，本地采用 `httpx` 获取网页源码，通过 BeautifulSoup 删除 `<script>`, `<style>`, `<header>`, `<nav>`, `<footer>` 等结构，抽离高价值纯净文字，并限制首部 `12,000` 字符作为 Extractor 的输入，规避长文本开销。

### 1.4 事实提取 (Extractor Agent)
* **核心类名**: `ExtractorAgent`
* **职责**:
  1. **结构化提取**: 接受具体网页正文，围绕给定的“子课题”与“当前查询词”，定位其中的核心事实与数据，并摘录支持该结论的原文段落作为“证据（evidence）”。
  2. **零样本拒绝**: 若该网页内容与当前子课题无关或纯属广告干扰，大模型应返回空 Fact 列表，确保数据池纯净。

### 1.5 报告合成 (Synthesizer Agent)
* **核心类名**: `SynthesizerAgent`
* **职责**:
  1. **事实整合与去重**: 汇聚全部的源文献及事实要点，按照子章节结构将多源信息归类。
  2. **Inline 引用绑定**: 分章节撰写学术化风格的报告，确保每处引用都有类似 `[1]`, `[2]` 的引用编号，并在报告末尾完整附带与其编号对应的 URL 列表。

## 2. 集中化 Prompt 配置管理

所有 Agent 的 Prompt 全集中在 `backend/app/prompts/prompts.yaml` 下，通过 `PromptManager` 实现模版化加载。

> [!CAUTION]
> **Python 字符串格式化冲突规避**:
> 因为大模型通常需要以规范的 JSON 格式输出结构化数据，这使得 System Prompt 内部必须定义大括号形式的 JSON Schema（例如 `{"sub_topics": [...]}`）。
> 为了避免 Python 在调用 `.format(**kwargs)` 时因解析这些非变量大括号而抛出 `KeyError`，**YAML 文件中所有的非变量大括号 `{}` 必须全部双写为 `{{}}`**。

### YAML 集中模版示例
* **Planner 提示词**:
  ```yaml
  planner:
    system: |
      你是一个专业的研究规划专家...
      你必须使用 JSON 格式响应，格式必须完全符合以下 JSON Schema：
      {{
        "sub_topics": [
          {{
            "title": "子方向标题",
            "reason": "拆解该方向的理由",
            "search_queries": ["查询词1", "查询词2"]
          }}
        ]
      }}
  ```
* **Extractor 提示词**:
  ```yaml
  extractor:
    system: |
      你是一个资深的事实提取专家...
      你必须使用 JSON 格式响应，格式必须为：
      {{
        "facts": [
          {{
            "fact": "提取的事实陈述",
            "evidence": "支持该事实的原文段落/论据/具体数字"
          }}
        ]
      }}
  ```
